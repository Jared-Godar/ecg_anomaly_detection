# v1.0.0 — First Portfolio Release (Retrospective)

> **This page is a retrospective.** v1.0.0 was tagged on 2026-07-08 with its own release notes,
> and the
> [published release body](https://github.com/Jared-Godar/ecg_anomaly_detection/releases/tag/v1.0.0)
> remains the honest historical record of what shipped. This page reframes that same release as a
> set of portfolio signals — what the first stable snapshot demonstrated as engineering evidence —
> without adding anything to it after the fact. Every claim below describes the repository as it
> stood at the `v1.0.0` tag.

This project began as a 2022 educational ECG classification experiment on the public MIT-BIH
Arrhythmia Database and was rebuilt from scratch as a governed, tested, reproducible
data-engineering case study. v1.0.0 was the moment that rebuild became a stable, reviewable
artifact: a configuration-driven local pipeline, an auditable lineage model, CI that executes the
public notebooks rather than merely inspecting them, a governance framework with real boundaries,
and a release process that audited its own claims before making them.

**Important use limitation:** this is not medical software. It is for research, education, and
software-engineering demonstration only — never for diagnosis, monitoring, treatment, or
patient-care decisions. It has not been clinically validated, and no release of this repository
implies production readiness, clinical suitability, medical utility, or regulatory compliance.

## A governed pipeline, not a notebook

The release shipped a complete configuration-driven local pipeline: versioned, fail-safe PhysioNet
acquisition; typed WFDB ingestion and per-record structural validation; versioned annotation
mapping; boundary-safe window extraction with explicit channel-identity resolution; deterministic
subject-aware splitting with split-quality diagnostics; deterministic baseline training; and
validation-only evaluation. Every stage was driven by versioned TOML configuration rather than
hard-coded choices, and the data-preparation stages were each exposed as their own CLI subcommand,
while deterministic baseline training and validation-only evaluation ran through the composite
`run-pipeline` orchestrator.

The load-bearing design decision was the split. The archived 2022 notebook divided individual beat
windows randomly after window creation, so windows from the same record could land on both sides
of the train/test boundary. The modern pipeline splits at the record/subject level instead,
retaining record identity as a grouping constraint — the defect the whole modernization was built
to eliminate, documented at the time in
[`docs/record-grouped-splitting.md`](../record-grouped-splitting.md).

## Lineage as a first-class artifact

Every pipeline run produced an auditable run manifest connecting Git state to generated outputs: a
UUID run identifier, the full commit hash plus a dirty-state flag, the Python implementation and
installed-package snapshot, and repository-relative path, byte size, and SHA-256 digest for every
referenced configuration, evidence, and artifact file. The manifest was framed honestly as
operational lineage for a reproducible data workflow — explicitly not a clinical validation record
and not a statement about model quality.

## Notebooks as tested software

The three curated public notebooks (environment setup and governed artifact generation, a
narrative walkthrough, and a validation-only gradient-boosting example) were executed end to end
in CI on every pull request — genuine cell-by-cell execution of the real, unmodified notebooks in
an isolated worktree seeded with synthetic WFDB records, so the real dataset was never downloaded.
This sat on top of a full quality gate: tests, Pyright type checking, formatting and linting,
secret scanning, and a package-build assurance check that builds a wheel and sdist without ever
publishing them — alongside automated weekly dependency updates.

## The boundary that was not crossed

v1.0.0's most distinctive signal is a deliberate absence. The supported evaluator scored only the
`validation` partition; the protected `test` partition was indexed but had never been opened or
scored. No held-out benchmark existed in the release, and the release notes said so in bold rather
than burying it. Defining and executing a held-out evaluation was intentionally deferred past 1.0
— because forcing a single irreversible, governance-gated action against a release deadline would
have contradicted the eligibility, approval, single-execution, and disclosure controls the project
had already written down in [`docs/benchmark-governance.md`](../benchmark-governance.md).

That arc was closed **after** v1.0.0, not in it: the governed held-out evaluation belongs to the
[v1.1.0 era](v1.1.0.md) and is recorded in
[`docs/held-out-evaluation.md`](../held-out-evaluation.md). Crediting it to this release would be
exactly the kind of claim inflation the governance exists to prevent.

## A claim audit with receipts

Before tagging, every model and pipeline claim in `README.md`, `MODEL_CARD.md`, and `docs/*` was
systematically audited against actual generated evidence, configuration, and code behavior. The
reproducibility claim was verified the hard way: a real `git clone` into a throwaway location,
from which the `dev`, `notebooks`, and `experiments` locked sync groups, the full test suite, and
Pyright all succeeded — a fresh-checkout result, not an assumption inherited from CI. Several
claims were additionally confirmed against live MIT-BIH source data. The release-time entry point
that audit verified is still the project's two-line quick start:

```fish
uv sync --locked --dev
uv run pytest
```

## What a release deliberately is not

Release governance ([`docs/governance/releases.md`](../governance/releases.md)) defined the
release as an engineering-portfolio artifact and nothing more: no dataset, derived patient data,
trained model, or benchmark output was part of it, and no package was published to PyPI or any
other index. Built `dist/` artifacts were confirmed untracked with `git ls-files dist` as part of
the [release checklist](../governance/release-checklist.md) — a boundary checked mechanically, not
asserted.

## The historical numbers, in their place

The archived 2022 notebook's saved metrics were preserved in the release as flagged historical
evidence only. Their random beat-window split could leak windows from the same record across
partitions, so they may be inflated and do not measure generalization to unseen subjects. They are
never presented as benchmarks or performance achievements — here or anywhere else in the
repository. The full audit of the historical outputs, including a known metric defect in the
original notebook, lives in [`docs/historical-results.md`](../historical-results.md).
